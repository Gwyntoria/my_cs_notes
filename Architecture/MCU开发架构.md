# MCU 开发架构

## Summary

MCU 工程的架构选择，核心不是「要不要上 RTOS」这一题，而是要同时回答以下问题：

1. 程序的执行流由谁驱动：主循环、定时节拍、事件、状态机，还是 RTOS 调度器。
2. 模块之间如何隔离：是否需要 BSP、HAL、Driver、Middleware、Application 这样的分层。
3. 并发和通信如何处理：轮询、标志位、消息队列、事件队列、回调、共享数据还是任务间同步。
4. 状态复杂度如何管理：简单分支、表驱动状态机、层次状态机，还是多个独立状态机。
5. 工程是否需要长期演进：是否涉及多硬件平台、多产品型号、OTA、诊断、日志、自动化测试和多人协作。

对于 MCU 工程，架构通常不是单选，而是组合。例如一个中等复杂度的设备，可能同时使用「分层架构 + 事件驱动 + 状态机 + 少量 RTOS 任务」；一个小型传感器节点，可能只需要「裸机 Super Loop + ISR 标志位 + 简单 Driver 分层」。

## 先区分几个维度

讨论 MCU 架构时，容易把不同层面的东西混在一起。建议拆成 5 个维度理解：

| 维度 | 关注点 | 常见选择 |
| :--- | :--- | :--- |
| 执行模型 | CPU 按什么顺序运行代码 | Super Loop、前后台架构、时间触发架构、事件驱动、RTOS 多任务 |
| 模块组织 | 代码如何分层和解耦 | BSP、HAL、Driver、Middleware、Service、Application |
| 通信机制 | 模块之间如何传递信息 | 全局变量、标志位、回调、事件队列、消息队列、发布订阅 |
| 状态管理 | 设备生命周期和业务流程如何表达 | if-else、switch-case、有限状态机、层次状态机、表驱动状态机 |
| 工程化机制 | 项目如何长期维护和量产 | 配置系统、日志、诊断、Bootloader、OTA、测试框架、平台抽象 |

## 常见 MCU 架构

### 裸机 Super Loop 架构

Super Loop 是最基础的 MCU 程序结构：初始化外设后进入 `while (1)`，在主循环里依次轮询各个模块。

```c
int main(void)
{
    board_init();
    app_init();

    while (1) {
        key_scan();
        sensor_poll();
        protocol_process();
        led_update();
    }
}
```

适用场景：

- 功能少，外设少，业务流程短。
- 对实时性要求不高，或者实时动作都能放在中断中完成。
- 团队人数少，生命周期短，后续变更不频繁。
- 典型项目：简单按键控制、LED 控制、小型传感器采集、单一串口命令设备。

优点：

- 理解成本低，调试简单，资源占用低。
- 不依赖 RTOS，启动快，移植成本低。
- 适合快速验证硬件和小型产品原型。

缺点：

- 一旦某个函数阻塞，整个系统响应都会变差。
- 模块之间容易通过全局变量耦合。
- 业务状态复杂后，主循环会变成难以维护的流程堆叠。

判断标准：

- 如果所有周期性任务都能在主循环中按时完成，并且最坏执行时间可接受，Super Loop 就可以成立。
- 如果需要同时处理通信、传感器、UI、低功耗、OTA 等多个异步流程，单纯 Super Loop 通常会变脆。

### 前后台架构

前后台架构 (Foreground/Background) 是在 Super Loop 基础上加入中断。中断处理紧急事件，主循环处理非紧急任务。

```c
volatile bool uart_rx_pending = false;

void USART_IRQHandler(void)
{
    uart_read_byte_to_buffer();
    uart_rx_pending = true;
}

int main(void)
{
    board_init();

    while (1) {
        if (uart_rx_pending) {
            uart_rx_pending = false;
            protocol_parse();
        }

        sensor_poll();
        led_update();
    }
}
```

适用场景：

- 需要响应外部中断，但后台业务仍然不复杂。
- 中断来源有限，例如 UART 接收、按键、定时器、ADC 完成中断。
- 数据量较小，ISR 只做搬运和置位，不在中断里做复杂业务。

优点：

- 比纯轮询响应更快。
- 资源占用仍然很低。
- 对小型裸机项目非常实用。

缺点：

- 中断和主循环之间容易出现共享数据竞争。
- 标志位增多后，主循环会变成大量 `if` 判断。
- 如果 ISR 做得太重，会影响系统实时性和可预测性。

实践原则：

- ISR 只做必要动作：读写寄存器、搬运数据、清中断、置标志、投递事件。
- 业务逻辑尽量放在主循环、任务或事件处理函数中。
- 中断和主循环共享的数据要考虑 `volatile`、临界区、环形缓冲区和原子性。

### 时间触发架构

时间触发架构 (Time-triggered Architecture) 以固定节拍驱动任务执行。常见做法是使用 SysTick 或硬件定时器产生系统 tick，再按周期调度不同任务。

```c
void scheduler_run(void)
{
    if (task_due(TASK_1MS)) {
        task_1ms();
    }

    if (task_due(TASK_10MS)) {
        task_10ms();
    }

    if (task_due(TASK_100MS)) {
        task_100ms();
    }
}

int main(void)
{
    board_init();

    while (1) {
        scheduler_run();
    }
}
```

适用场景：

- 任务以固定周期为主，事件突发性不强。
- 需要较强的时序可预测性。
- 控制类、采样类、周期通信类工程。
- 典型项目：电机控制外围逻辑、周期采样设备、简单 BMS、工业传感器、仪表采集。

优点：

- 时序关系清晰，容易计算周期和负载。
- 便于做最坏执行时间分析。
- 不一定需要 RTOS。

缺点：

- 对突发事件处理不如事件驱动自然。
- 任务执行时间超预算时，会影响后续任务。
- 不适合大量不确定事件和复杂交互流程。

实践原则：

- 每个周期任务都要有执行时间预算。
- 低频任务不能阻塞高频任务。
- 定时器中断只更新时间基准，不直接执行重业务。

### 事件驱动架构

事件驱动架构 (Event-driven Architecture) 把外设输入、通信输入、定时器到期、状态变化等都抽象成事件。系统主循环或调度器不断取事件并分发给处理函数。

```c
typedef enum {
    EVENT_KEY_PRESSED,
    EVENT_UART_FRAME_RECEIVED,
    EVENT_SENSOR_READY,
    EVENT_TIMEOUT,
} event_type_t;

typedef struct {
    event_type_t type;
    uint32_t param;
} event_t;

int main(void)
{
    board_init();
    event_queue_init();

    while (1) {
        event_t event;

        if (event_queue_pop(&event)) {
            app_dispatch_event(&event);
        }
    }
}
```

适用场景：

- 外部输入多，业务由事件触发。
- UI、按键、通信协议、无线连接、传感器上报、超时处理交织在一起。
- 希望降低模块之间直接调用关系。
- 典型项目：BLE 设备、低功耗传感器、智能家居节点、交互式控制器。

优点：

- 能把「发生了什么」和「如何处理」分开。
- 模块之间可以通过事件通信，减少直接依赖。
- 适合和状态机组合使用。

缺点：

- 事件定义不清晰时，会变成另一种形式的全局耦合。
- 调试时需要追踪事件来源、队列顺序和处理耗时。
- 队列长度、事件丢弃策略、优先级策略需要提前设计。

实践原则：

- 事件名描述事实，不描述动作。例如使用 `EVENT_UART_FRAME_RECEIVED`，不要使用 `EVENT_DO_PARSE_UART`。
- 事件处理函数不要长时间阻塞。
- 明确事件队列满时的策略：丢弃新事件、覆盖旧事件、计数合并还是触发错误处理。

### 状态机架构

状态机架构用明确的状态和状态迁移表达业务生命周期。它通常不是单独存在，而是和 Super Loop、事件驱动或 RTOS 任务组合。

```c
typedef enum {
    DEVICE_STATE_INIT,
    DEVICE_STATE_IDLE,
    DEVICE_STATE_CONNECTING,
    DEVICE_STATE_WORKING,
    DEVICE_STATE_ERROR,
} device_state_t;

void device_handle_event(event_t *event)
{
    switch (device_state) {
    case DEVICE_STATE_IDLE:
        if (event->type == EVENT_START) {
            device_state = DEVICE_STATE_CONNECTING;
        }
        break;

    case DEVICE_STATE_WORKING:
        if (event->type == EVENT_ERROR) {
            device_state = DEVICE_STATE_ERROR;
        }
        break;

    default:
        break;
    }
}
```

适用场景：

- 设备有明确生命周期。
- 业务流程存在模式切换、超时、错误恢复、重试、连接断开重连。
- 同一个事件在不同状态下行为不同。
- 典型项目：蓝牙连接设备、联网设备、充电设备、门锁、可穿戴设备、仪器工作流程。

优点：

- 状态边界清晰，异常流程更容易补全。
- 比大量分散的标志位更可维护。
- 适合写测试用例和做故障注入。

缺点：

- 状态拆分过细会导致状态爆炸。
- 多个状态机互相影响时，需要设计事件边界和状态归属。
- 表驱动状态机可读性依赖命名和文档质量。

实践原则：

- 先画状态图，再写代码。
- 每个状态要明确入口动作、运行逻辑、退出动作和可接受事件。
- 避免多个模块直接修改同一个状态变量。

### 协作式调度架构

协作式调度架构介于裸机和 RTOS 之间。系统维护一组任务函数，每个任务主动让出执行权，不发生抢占式上下文切换。

适用场景：

- 不想引入 RTOS，但 Super Loop 已经开始混乱。
- 任务数量适中，任务都能拆成短步骤。
- 系统需要一定结构化调度，但资源比较紧张。

优点：

- 比 RTOS 更轻量。
- 不存在抢占导致的多数竞态问题。
- 任务调度点明确，调试相对简单。

缺点：

- 任一任务不主动让出执行权，系统仍然会卡住。
- 不适合复杂阻塞等待。
- 对开发者编码纪律要求高。

### RTOS 多任务架构

RTOS 多任务架构使用实时操作系统调度多个任务。典型机制包括任务、优先级、队列、信号量、互斥锁、事件组、软件定时器。

适用场景：

- 存在多个并发活动，并且它们的优先级、周期、阻塞等待关系不同。
- 通信协议、文件系统、网络、UI、传感器、控制逻辑同时存在。
- 第三方协议栈或 SDK 已经依赖 RTOS。
- 典型项目：复杂 BLE/Wi-Fi 设备、网关类 MCU、带显示屏的控制器、需要 OTA 的量产设备、多协议设备。

优点：

- 并发模型清晰，适合处理阻塞等待。
- 可以通过优先级表达实时性差异。
- 更容易集成复杂中间件和协议栈。

缺点：

- 任务切换、栈、内核对象会消耗 RAM 和 Flash。
- 优先级反转、死锁、竞态、栈溢出等问题需要工程经验。
- 如果任务边界设计不好，会把简单问题复杂化。

实践原则：

- 不要按「模块」机械创建任务，而要按「并发需求」创建任务。
- 高优先级任务必须短小、可预测，避免长时间持锁。
- 任务之间优先用消息队列传递数据，少用共享全局状态。
- 所有任务栈大小都要有测量依据，不能只靠经验值。

### 分层架构

分层架构关注代码边界，而不是 CPU 如何调度。典型 MCU 工程可以分为：

```text
Application
Service / Middleware
Driver
HAL
BSP
CMSIS / Vendor SDK
Hardware
```

各层职责：

| 层级 | 职责 |
| :--- | :--- |
| BSP | 板级初始化、时钟、引脚、外设资源映射 |
| HAL | 屏蔽芯片寄存器和厂商库差异，提供稳定硬件接口 |
| Driver | 面向具体器件的驱动，例如 IMU、Flash、Sensor、Motor Driver |
| Middleware | 协议栈、文件系统、日志、参数存储、升级、加密 |
| Service | 面向产品能力的服务，例如电源管理、连接管理、采集服务 |
| Application | 业务流程、产品状态机、用户交互、策略决策 |

适用场景：

- 工程需要长期维护。
- 可能更换 MCU、板卡、传感器或通信模块。
- 多人协作，需要清晰模块边界。
- 同一套业务要复用到多个产品型号。

优点：

- 可移植性和可测试性更好。
- 底层变更对上层影响可控。
- 便于做代码审查、模块交接和多人协作。

缺点：

- 小项目中过度分层会增加样板代码。
- 层次边界不清时，会出现跨层调用和反向依赖。
- HAL 设计过厚会掩盖硬件特性，过薄又无法提供稳定抽象。

实践原则：

- Application 不直接操作寄存器。
- Driver 不依赖具体业务状态。
- BSP 负责「这块板子怎么连」，不负责「产品应该怎么工作」。
- HAL 只抽象稳定能力，不强行统一所有芯片差异。

### 组件化或服务化架构

组件化架构把系统拆成相对独立的能力单元，每个组件有自己的初始化、配置、事件处理和对外接口。

示例组件：

- `power_manager`
- `sensor_manager`
- `connection_manager`
- `storage_service`
- `ota_service`
- `diagnostic_service`
- `log_service`

适用场景：

- 产品能力较多，模块需要复用。
- 多个产品共享同一套基础能力。
- 团队多人开发，需要明确 owner 和接口。
- 典型项目：可穿戴设备、智能家居产品、工业控制器、医疗或仪器类 MCU 产品。

优点：

- 模块复用和替换更容易。
- 更适合做单元测试和集成测试。
- 便于形成团队内部平台代码。

缺点：

- 需要统一生命周期管理和配置方式。
- 组件之间可能形成隐式依赖链。
- 对事件、错误码、日志、配置、资源管理的一致性要求更高。

实践原则：

- 每个组件都要有明确的公开接口和私有状态。
- 组件初始化顺序要显式化。
- 组件之间尽量通过接口、事件或消息通信，不直接访问彼此内部变量。

### Active Object 架构

Active Object 可以理解为「对象 + 私有事件队列 + 独立执行上下文」。在 MCU 中，它可以运行在裸机事件循环上，也可以映射到 RTOS 任务。

适用场景：

- 系统由多个相对独立的状态机组成。
- 每个对象都需要异步接收事件，但不希望共享大量状态。
- 希望比传统 RTOS 任务更强调事件和状态边界。

典型例子：

- 连接对象负责配对、连接、断连和重连。
- 采集对象负责传感器启动、采样、滤波和上报。
- 电源对象负责充电、休眠、唤醒和电量状态。

优点：

- 状态封装好，模块之间耦合低。
- 适合复杂交互和多个状态机协作。
- 可以减少共享内存带来的竞态。

缺点：

- 事件设计和对象边界需要较强经验。
- 调试时要跟踪多个事件队列。
- 对极小工程来说抽象成本偏高。

### Bootloader + Application 架构

Bootloader + Application 不是业务执行模型，而是量产、升级和维护架构。Bootloader 负责启动校验、固件选择、升级恢复和跳转应用。

适用场景：

- 产品需要 OTA 或有线升级。
- 设备部署后不容易人工拆机烧录。
- 需要防止升级失败导致设备变砖。
- 需要版本管理、固件校验、回滚或安全启动。

常见设计点：

- Flash 分区：Bootloader、Application、升级包、参数区、日志区。
- 固件校验：CRC、Hash、签名校验。
- 启动策略：单 Bank、双 Bank、A/B 分区、差分升级。
- 失败恢复：升级中断、校验失败、应用启动失败时如何回退。

判断标准：

- 只要产品有现场升级需求，Bootloader 就应该尽早纳入架构，而不是后期补丁式加入。
- 如果设备有安全要求，固件签名和防回滚策略应在 Flash 分区设计阶段一起考虑。

### 诊断和可观测性架构

诊断架构关注系统出问题时如何定位，而不是正常流程如何跑通。它包括日志、错误码、事件记录、崩溃信息、运行统计、通信抓取和现场诊断命令。

适用场景：

- 产品需要量产和售后。
- 设备部署在现场，复现问题成本高。
- 通信、低功耗、传感器或时序问题较多。
- 团队需要从「能跑」进入「能定位问题」阶段。

常见能力：

- 分级日志：`ERROR`、`WARN`、`INFO`、`DEBUG`。
- 错误码体系：模块号、错误类型、上下文参数。
- 运行统计：任务水位、队列水位、重启原因、内存使用、看门狗次数。
- 关键事件记录：连接、断连、升级、异常、低电、传感器错误。
- 现场命令：查询版本、读取参数、导出日志、执行自检。

判断标准：

- 如果项目已经进入量产或多人联调阶段，没有诊断架构会显著增加定位成本。
- 如果设备没有屏幕和文件系统，至少应保留串口日志、错误码和重启原因记录。

### 低功耗架构

低功耗架构关注系统如何在不同功耗状态之间切换，以及哪些模块有权阻止睡眠。

适用场景：

- 电池供电设备。
- 需要长时间待机。
- 唤醒源多，例如按键、RTC、传感器中断、无线事件。
- 典型项目：可穿戴设备、遥控器、传感器节点、BLE Beacon。

核心问题：

- 系统有哪些功耗状态：运行、空闲、浅睡眠、深睡眠、关机。
- 每个状态下哪些外设保持工作。
- 哪些事件可以唤醒系统。
- 进入睡眠前是否需要保存上下文。
- 唤醒后是冷启动、热启动还是恢复运行。

实践原则：

- 低功耗不是最后加一个 `sleep()`，而是贯穿外设、时钟、任务、事件和状态机设计。
- 每个模块应声明自己的电源需求，例如「正在写 Flash」「正在传输」「正在采样」时不能进入深睡眠。
- 电源管理模块应集中决策，避免各模块直接随意切换系统电源状态。

## 架构适用工程体量

| 工程类型 | 典型特征 | 推荐架构组合 |
| :--- | :--- | :--- |
| Demo 或硬件验证 | 功能单一，生命周期短，主要验证外设 | Super Loop + 简单 Driver |
| 小型控制器 | 少量外设，少量中断，业务流程简单 | 前后台架构 + 简单分层 |
| 周期采集或控制设备 | 任务周期明确，需要稳定采样或控制节奏 | 时间触发架构 + Driver 分层 + 状态机 |
| 交互式设备 | 按键、显示、通信、超时事件较多 | 事件驱动 + 状态机 + 分层架构 |
| 中型量产设备 | 多外设、多状态、需要升级和诊断 | 分层架构 + 事件驱动 + Bootloader + 诊断架构 |
| 复杂连接设备 | BLE、Wi-Fi、协议栈、OTA、文件系统并存 | RTOS 多任务 + 消息队列 + 组件化架构 |
| 多产品线平台 | 多板卡、多型号、多团队复用 | HAL/BSP 抽象 + 组件化 + 配置系统 + 自动化测试 |
| 强实时或安全相关设备 | 时序严格，错误后果严重，需要可预测性 | 时间触发架构 + 静态资源分配 + 状态机 + 严格诊断 |

## 选型判断路径

可以按下面顺序判断：

1. 如果功能少、事件少、没有复杂阻塞等待，优先考虑 Super Loop 或前后台架构。
2. 如果系统主要由固定周期任务构成，优先考虑时间触发架构。
3. 如果系统主要由外部输入和异步流程驱动，优先考虑事件驱动架构。
4. 如果设备生命周期复杂，必须引入状态机。
5. 如果存在多个长期并发活动，并且它们有不同优先级或阻塞等待需求，再考虑 RTOS。
6. 如果项目要量产、升级、售后和多人维护，必须补上分层、诊断、Bootloader 和配置管理。

一个务实的判断是：先证明裸机架构不够用，再引入 RTOS；但一旦已经明确存在复杂协议栈、多阻塞流程、多优先级实时任务，就不要为了「简单」硬撑裸机。

## 提问清单

### 产品和需求

- 这个 MCU 在整个产品里承担什么职责：主控、协处理器、通信节点、传感器节点还是执行器控制器？
- 设备是一次性 Demo、内部工具、量产产品，还是平台型产品？
- 项目的预期生命周期是几周、几个月，还是几年？
- 后续是否会派生多个硬件版本或产品型号？
- 失败后果是什么：只是功能异常、用户体验下降、设备不可用，还是存在安全风险？

### 实时性和时序

- 系统里最严格的响应时间是多少？是微秒级、毫秒级，还是秒级？
- 哪些任务必须按固定周期运行？
- 哪些任务可以延迟处理？
- 是否能列出每个周期任务的最坏执行时间？
- 是否有任务会长时间阻塞，例如等待通信、等待 Flash 写入、等待传感器转换？
- 中断频率最高的来源是什么？中断里最多能做多少工作？

### 并发和事件

- 系统同时存在多少条独立活动线？例如采集、通信、显示、存储、升级、控制。
- 这些活动是否需要同时等待不同事件？
- 模块之间是同步调用为主，还是异步通知为主？
- 是否需要事件优先级？
- 队列满、消息丢失、事件重复到达时怎么处理？
- 是否有共享资源，例如 UART、SPI、I2C、Flash、日志输出、参数区？

### 状态复杂度

- 设备有哪些主要状态？例如初始化、空闲、连接中、工作中、升级中、错误、休眠。
- 每个状态允许响应哪些事件？
- 同一个事件在不同状态下是否有不同行为？
- 是否存在超时、重试、回滚、错误恢复？
- 状态是否由一个模块统一管理，还是散落在多个标志位里？
- 状态切换是否需要记录原因，方便调试和售后分析？

### 硬件资源

- MCU 的 Flash、RAM、主频、外设数量是否充足？
- RTOS 任务栈和内核对象会消耗多少 RAM？
- 是否允许动态内存分配？如果允许，谁负责失败处理和碎片风险？
- Flash 是否需要分区给 Bootloader、应用、升级包、参数和日志？
- 是否有 DMA、低功耗定时器、硬件 CRC、加密单元等可用于架构简化的资源？

### 通信和协议

- 通信协议是简单命令式，还是存在连接、鉴权、分包、重传、加密、心跳？
- 是否需要同时处理多个通信接口？
- 协议栈是否由第三方 SDK 提供？它是否要求 RTOS？
- 协议解析是否能在主循环中完成，还是需要独立任务或事件队列？
- 通信异常后如何恢复？是否需要断线重连和状态同步？

### 低功耗

- 设备是否电池供电？
- 待机电流、工作电流和续航目标分别是多少？
- 系统有哪些睡眠状态？
- 哪些模块可以阻止睡眠？
- 唤醒源有哪些？
- 唤醒后要恢复哪些状态？
- RTOS tick、定时器、调试日志是否会影响低功耗？

### 升级和量产

- 产品是否需要 OTA 或现场升级？
- 升级失败后如何恢复？
- 是否需要双分区、回滚、固件校验或签名？
- 参数区是否需要版本迁移？
- 量产时如何写入 SN、校准参数、密钥和出厂配置？
- 售后如何确认设备当前固件版本、重启原因和错误历史？

### 可测试性和可维护性

- 业务逻辑是否能脱离硬件做单元测试？
- Driver、HAL、Application 之间边界是否清晰？
- 是否有统一错误码、日志接口和断言策略？
- 是否能模拟传感器数据、通信输入和异常条件？
- 新人是否能通过目录结构理解模块职责？
- 修改一个传感器型号时，影响范围是否只在 Driver 或 HAL 层？

### 团队和协作

- 代码由一个人维护，还是多人并行开发？
- 是否需要模块 owner？
- 是否需要统一接口规范、命名规范和目录结构？
- 是否有代码审查、静态检查、自动化构建和版本发布流程？
- 架构文档是否能解释模块边界、启动流程、任务关系和状态机？

## 常见误区

### 误区一：上了 RTOS 就等于架构清晰

RTOS 只解决调度问题，不自动解决模块边界、状态管理和资源 ownership。一个混乱的裸机工程，直接搬到 RTOS 上，通常会变成多个混乱任务互相等待和共享全局变量。

### 误区二：小项目一定不需要架构

小项目不需要重架构，但仍然需要基本边界。例如 `board_init()`、`driver_xxx`、`app_xxx` 的分离，可以显著降低后续修改成本。

### 误区三：事件越多越灵活

事件驱动的关键是事件语义清晰。事件过细会导致调度碎片化；事件过粗会导致处理函数内部继续堆业务判断。

### 误区四：状态机只是 `switch-case`

`switch-case` 只是实现方式。真正重要的是状态定义、事件定义、迁移条件、入口动作、退出动作和异常恢复策略。

### 误区五：低功耗可以最后再做

低功耗会影响时钟、外设、中断、任务阻塞、日志、通信和状态恢复。对于电池设备，低功耗应该在架构阶段就进入设计。

## 推荐落地方式

1. 先画一张系统上下文图：MCU 和传感器、执行器、通信模块、服务器、手机 App、PC 工具之间的关系。
2. 再画一张软件分层图：BSP、HAL、Driver、Middleware、Service、Application 分别有什么模块。
3. 列出所有周期任务和异步事件。
4. 列出设备主状态机。
5. 判断是否需要 RTOS：只根据并发、阻塞、优先级和第三方 SDK 依赖判断，不根据「项目看起来高级」判断。
6. 在架构早期决定日志、错误码、参数存储、升级和低功耗策略。
7. 写一页架构说明，至少包含启动流程、任务关系、事件流、状态机、Flash 分区和关键资源 ownership。
