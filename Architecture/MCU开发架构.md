# MCU 常用嵌软开发架构

## 1. 这篇文章解决什么问题

MCU 嵌软架构设计常见的误区是：一谈架构，就直接讨论要不要 RTOS、要不要 C++ 抽象类、要不要设计模式。

但在实际工程里，架构选型首先要回答的是：

- 这个工程的主要复杂度来自哪里？
- 这个复杂度是时序复杂度、状态复杂度、并发复杂度，还是复用复杂度？
- 当前芯片资源、团队能力、产品生命周期是否支撑这套架构？
- 这个架构能否让调试、移植、联调、量产维护变得更可控？

MCU 工程和 PC/Server 软件不同，约束通常更硬：

- Flash、RAM、主频、外设数量有限；
- 中断响应、采样周期、控制周期经常有确定性要求；
- 调试和日志能力有限，很多问题只能在线复现；
- GPIO、UART、SPI、I2C、ADC、PWM、DMA、Timer 等外设直接参与业务闭环；
- 量产后维护周期长，芯片替换、版本分支、产品变体都很常见。

因此，MCU 架构设计的核心不是“看起来高级”，而是：

- 让实时路径可分析；
- 让业务状态可表达；
- 让模块依赖可控制；
- 让硬件差异可隔离；
- 让后续扩展和维护不失控。

---

## 2. 先看总体分类

MCU 上常见架构可以按工程成熟度理解为 7 类：

| 架构类型 | 核心特征 | 主要解决的问题 | 典型体量 |
| --- | --- | --- | --- |
| 裸机顺序式 | `while (1)` 顺序轮询 | 快速跑通单一功能 | 很小型 |
| 中断驱动 + 前后台 | ISR 处理快事件，主循环处理慢业务 | 异步输入和实时响应 | 小型到中小型 |
| 时间片轮询 / 协作式调度 | 主循环按节拍调度多个周期任务 | 多个轻量周期任务 | 小型到中型 |
| 模块化分层 | BSP、Driver、Service、App 分层 | 驱动和业务解耦 | 中型 |
| 事件驱动 / 状态机 | 事件触发状态迁移 | 状态和流程复杂度 | 中型到中大型 |
| RTOS 任务化 | 多 task、队列、信号量、优先级 | 并发和阻塞复杂度 | 中大型 |
| 平台化 / 组件化 | 接口、配置、组件、产品变体 | 多产品复用和移植 | 大型或产品线 |

这些类型不是互斥关系。一个成熟 MCU 项目经常是叠加形态：

```text
中断快路径
    + 分层驱动
    + 事件队列
    + 状态机
    + 必要时使用 RTOS
    + 产品化后加入组件化和配置化
```

所以选型时不应问“我应该选哪一个架构”，而应问：

- 哪些部分需要中断快路径？
- 哪些部分适合后台轮询？
- 哪些部分需要状态机？
- 哪些部分需要 task？
- 哪些部分需要被抽象成可替换组件？

## 3. 架构一：裸机顺序式

### 3.1 典型形态

最简单的 MCU 程序通常是：

```c
int main(void)
{
    board_init();
    driver_init();

    while (1)
    {
        key_scan();
        adc_sample();
        control_calc();
        led_update();
        uart_poll();
    }
}
```

它的特点是：

- 初始化后进入无限循环；
- 所有任务在 `while (1)` 中顺序执行；
- 中断即使存在，也只做非常少的辅助工作；
- 整个系统控制流非常直接。

### 3.2 适合什么工程

适合：

- 功能很少的单板程序；
- 快速验证、Demo、实验代码；
- 一次性测试工具；
- 简单传感器采集；
- 简单按键、LED、继电器控制；
- RAM/Flash 很紧的小 MCU。

不适合：

- 有复杂通信协议；
- 有多个相互独立的功能域；
- 有明显异步事件；
- 有较多状态切换和异常恢复；
- 需要长期维护和复用。

### 3.3 工程体量判断

如果满足大部分条件，可以使用裸机顺序式：

- 主循环里只有几个函数；
- 每个函数执行时间可控；
- 没有阻塞等待；
- 没有复杂状态机；
- 没有多路通信并发；
- 后续需求增长有限。

一旦 `while (1)` 里开始堆大量 `if/else`、flag、延时函数和协议解析，就说明它已经开始不适合。

### 3.4 主要风险

- 一处函数耗时会拖慢所有逻辑；
- 模块之间容易互相访问全局变量；
- 主循环越来越臃肿；
- 实时性边界难分析；
- 需求增长后容易变成面条代码。

## 4. 架构二：中断驱动 + 前后台

### 4.1 典型形态

这是 MCU 项目中非常常见的架构。

```c
volatile uint8_t uart_rx_flag = 0;

void USART_IRQHandler(void)
{
    uart_rx_byte = USARTx->DR;
    uart_rx_flag = 1;
}

int main(void)
{
    system_init();

    while (1)
    {
        if (uart_rx_flag)
        {
            uart_rx_flag = 0;
            protocol_parse();
        }

        key_scan();
        app_update();
    }
}
```

系统被分成两部分：

- 前台：中断服务程序 ISR，处理高实时事件；
- 后台：主循环，处理低实时业务。

### 4.2 核心原则

中断里只做短、快、确定的动作：

- 读写寄存器；
- 搬运一个字节或一段 DMA 数据；
- 记录时间戳；
- 写 ring buffer；
- 置位事件 flag；
- 触发轻量通知。

不要在中断里做：

- 复杂协议解析；
- 阻塞等待；
- 大量日志打印；
- 文件系统访问；
- 动态内存分配；
- 不确定耗时的业务逻辑。

### 4.3 适合什么工程

适合：

- 有 UART、CAN、SPI、I2C、ADC、Timer 等异步事件；
- 业务不算复杂，但实时性已经重要；
- 控制逻辑主要由定时器或采样中断驱动；
- 后台处理可以按事件慢慢消费；
- 工业控制、小家电、仪器仪表、简单通信设备。

不适合：

- 后台任务之间存在大量阻塞等待；
- 多个功能域需要独立优先级；
- 协议栈、UI、存储、联网等复杂模块同时存在；
- 主循环已经很难安排执行顺序。

### 4.4 常见配套结构

前后台架构通常会配合：

- `volatile` flag；
- ring buffer；
- 事件位；
- 简单消息队列；
- 定时器节拍；
- 状态机；
- DMA 双缓冲。

### 4.5 主要风险

- 中断和主循环共享变量没有考虑原子性；
- ISR 越写越长；
- flag 太多后事件关系不清晰；
- 后台处理速度跟不上前台事件产生速度；
- 中断优先级设计混乱。

## 5. 架构三：时间片轮询 / 协作式调度

### 5.1 典型形态

当裸机项目里周期性任务变多，可以在不上 RTOS 的情况下引入轻量调度。

```c
int main(void)
{
    system_init();

    while (1)
    {
        scheduler_run();
    }
}
```

调度器根据系统节拍执行任务：

```c
static task_t tasks[] = {
    { key_scan,      10  },
    { sensor_update, 20  },
    { control_loop,  5   },
    { led_update,    100 },
    { comm_poll,     1   },
};
```

它不是抢占式调度，而是协作式调度：

- 每个任务主动运行一小段；
- 任务不能长时间阻塞；
- 调度器只决定何时调用；
- 实时性仍主要靠中断和任务执行时间控制。

### 5.2 适合什么工程

适合：

- 有多个固定周期任务；
- 周期任务执行时间都较短；
- 不需要真正多线程；
- 任务之间共享数据较简单；
- 想比手写 `if (tick)` 更规范；
- 资源不足以引入 RTOS。

典型例子：

- 采集周期 10 ms；
- 控制周期 5 ms；
- 按键扫描 20 ms；
- LED 刷新 100 ms；
- 通信轮询 1 ms。

### 5.3 和 RTOS 的区别

| 对比项 | 协作式调度 | RTOS |
| --- | --- | --- |
| 调度方式 | 主循环主动调用 | 抢占或时间片 |
| 栈开销 | 通常共用主栈 | 每个 task 独立栈 |
| 阻塞等待 | 不适合 | 适合 |
| 优先级表达 | 较弱 | 较强 |
| 调试复杂度 | 较低 | 较高 |
| 适用体量 | 小型到中型 | 中大型 |

### 5.4 主要风险

- 某个任务耗时过长会拖慢所有任务；
- 如果任务内部阻塞，整个系统都会卡住；
- 周期和优先级关系需要人工设计；
- 任务数量继续增长后会接近“手写 RTOS”。

## 6. 架构四：模块化分层

### 6.1 典型分层

很多 MCU 项目真正需要的第一层架构能力，不是 RTOS，而是分层。

常见分层如下：

```text
Application
Service / Middleware
Driver
BSP / Board
Chip / HAL
```

目录形态可以是：

```text
project/
├── app/
├── services/
├── middleware/
├── drivers/
├── bsp/
├── port/
└── main.c
```

### 6.2 各层职责

| 层级 | 职责 | 不应该做什么 |
| --- | --- | --- |
| `Chip/HAL` | 芯片寄存器、厂家库、底层外设适配 | 不写产品业务 |
| `BSP/Board` | 板级初始化、引脚、时钟、电源、外设实例 | 不直接承载复杂业务流程 |
| `Driver` | 外设驱动、设备驱动、基础读写能力 | 不反向依赖 App |
| `Middleware` | 协议栈、文件系统、滤波、队列、通用组件 | 不绑定具体产品策略 |
| `Service` | 参数、日志、诊断、通信服务、设备管理 | 不直接散落硬件细节 |
| `Application` | 产品业务、状态流转、控制策略 | 不直接操作大量寄存器 |

### 6.3 推荐依赖方向

尽量保持单向依赖：

```text
Application -> Service -> Middleware -> Driver -> BSP -> Chip
```

如果底层需要通知上层，优先使用：

- 回调；
- 事件；
- 消息队列；
- 观察者注册；
- 接口函数表。

避免：

- Driver 直接调用 App；
- Service 到处访问寄存器；
- App 绕过 Driver 直接操作硬件；
- 各模块横向互相调用；
- 全局变量成为事实上的接口。

### 6.4 适合什么工程

适合：

- 有多个外设；
- 有参数管理、存储、协议、诊断、控制策略；
- 项目会长期维护；
- 需要移植到不同板卡；
- 需要多人协作；
- 需要把硬件代码和业务代码分开演进。

### 6.5 主要价值

分层架构解决的是“变化隔离”：

- 换芯片时，尽量只动 `Chip/HAL` 和 `BSP`；
- 换板子时，尽量只动 `BSP` 和部分 `Driver`；
- 换业务策略时，尽量只动 `Application`；
- 换通信协议时，尽量只动 `Middleware` 或 `Service`。

如果没有先分层，即使引入 RTOS，也只是把混乱代码分散到多个 task 里。

## 7. 架构五：事件驱动 / 状态机

### 7.1 为什么需要状态机

很多 MCU 项目变复杂，不是因为计算量大，而是因为状态和事件关系复杂。

常见例子：

- 按键短按、长按、双击；
- 设备待机、运行、故障、升级、恢复；
- 通信握手、超时、重发、异常恢复；
- 电机启动、加速、稳态、减速、刹车；
- 充电、放电、保护、校准、告警。

如果不用状态机，这些逻辑很容易散落成大量 `if/else`。

### 7.2 核心模型

状态机把业务拆成：

- 状态：系统当前处于什么阶段；
- 事件：外部或内部发生了什么；
- 条件：事件是否允许触发迁移；
- 动作：迁移时执行什么；
- 下一个状态：动作之后进入哪里。

例如：

```text
状态：IDLE、RUNNING、FAULT、UPGRADE
事件：按键按下、采样完成、通信命令、超时、故障触发
动作：启动输出、关闭电机、保存参数、回复上位机
```

### 7.3 常见实现方式

| 实现方式 | 适用情况 |
| --- | --- |
| `switch-case` 状态机 | 状态少、逻辑直观 |
| 表驱动状态机 | 状态迁移规则比较规整 |
| 分层状态机 HSM | 状态有父子层级，共同行为多 |
| 事件队列 + 状态处理器 | 事件来源多，需要统一消费 |
| active object | 每个业务对象有自己的事件队列和状态 |

### 7.4 适合什么工程

适合：

- 设备模式较多；
- 错误恢复流程较多；
- 协议存在握手、超时、重试；
- UI 页面和输入事件较复杂；
- 需要把事件、状态、动作分开；
- 需要画状态图和做评审。

### 7.5 和 RTOS 的关系

状态机和 RTOS 不是替代关系。

- 状态机解决流程复杂度；
- RTOS 解决并发和调度复杂度。

很多项目可以是：

```text
裸机 + 事件队列 + 状态机
```

也可以是：

```text
RTOS task + 消息队列 + 状态机
```

如果复杂度主要来自“状态多、事件多、流程多”，优先考虑状态机，而不是第一时间上 RTOS。

## 8. 架构六：RTOS 任务化

### 8.1 RTOS 解决什么问题

RTOS 的价值不是让代码显得更高级，而是解决：

- 多个相对独立任务需要并发执行；
- 不同任务有不同优先级；
- 某些任务会阻塞等待事件；
- 需要统一队列、信号量、互斥锁、事件组；
- 需要软件定时器和线程间通信；
- 系统时序复杂到主循环难以维护。

### 8.2 典型任务划分

一个中大型 MCU 产品可能拆成：

| task | 职责 |
| --- | --- |
| `control_task` | 控制算法、闭环计算、输出更新 |
| `comm_task` | UART、CAN、以太网、BLE、Wi-Fi 通信 |
| `ui_task` | 按键、屏幕、蜂鸣器、LED |
| `storage_task` | 参数保存、日志、文件系统 |
| `monitor_task` | 故障检测、喂狗、运行统计 |
| `upgrade_task` | Bootloader 交互、OTA、固件升级 |

任务划分应该按职责域，而不是按函数数量。

### 8.3 适合什么工程

适合：

- 通信、控制、UI、存储、诊断等功能并存；
- 协议栈复杂，例如 TCP/IP、USB、BLE、Wi-Fi；
- 某些模块天然阻塞，例如等待网络、等待文件系统、等待消息；
- 不同功能的实时优先级差异明显；
- 中间件本身要求 RTOS；
- 项目未来会持续扩展。

典型项目：

- 带屏幕的人机交互设备；
- IoT 终端；
- 工业网关；
- 协议转换器；
- 复杂控制器；
- 带文件系统、USB、网络协议栈的设备。

### 8.4 不适合什么工程

不适合，或至少不急着使用：

- 功能很少；
- 任务数量少；
- 没有明显阻塞场景；
- 控制周期很短且路径必须高度确定；
- RAM 很紧；
- 团队缺少 RTOS 调试经验；
- 分层和状态机都还没有做好。

### 8.5 RTOS 的代价

引入 RTOS 会带来：

- 每个 task 都需要独立栈；
- 抢占导致时序分析更复杂；
- 共享资源需要锁和临界区；
- 可能出现死锁、竞态、优先级反转；
- 调试需要理解调度器行为；
- task 数量失控后系统会更难维护。

### 8.6 关键认知

实时性不等于 RTOS。

很多硬实时路径仍然应该由：

- 中断；
- DMA；
- Timer；
- ADC/PWM 硬件触发；
- 短路径控制回路；
- 严格的临界区控制。

RTOS 更擅长管理复杂任务，不应该替代底层硬实时机制。

## 9. 架构七：平台化 / 组件化

### 9.1 什么时候需要平台化

当工程不再是单个项目，而是出现：

- 多个产品型号；
- 多块板卡；
- 多种芯片平台；
- 多个团队协作；
- 需要复用通信、升级、日志、参数、诊断能力；
- 需要长期维护多个版本分支；

架构重点就会从“单项目能跑”变成“多项目可复用”。

### 9.2 常见结构

```text
products/
├── product_a/
├── product_b/
└── product_c/

app/
services/
components/
interfaces/
drivers/
platform/
├── stm32/
├── gd32/
└── nrf52/
bsp/
```

典型分工：

| 模块 | 作用 |
| --- | --- |
| `products` | 产品配置、功能裁剪、板级选择 |
| `interfaces` | 硬件和服务抽象接口 |
| `components` | 可复用组件，例如 ring buffer、状态机、日志 |
| `services` | 参数、升级、诊断、通信服务 |
| `platform` | 芯片平台适配 |
| `bsp` | 具体板卡适配 |

### 9.3 常见能力

平台化通常需要：

- 统一接口规范；
- 配置系统；
- 模块裁剪机制；
- 编译期产品选择；
- 单元测试或仿真桩；
- 自动化构建；
- Bootloader + Application 升级框架；
- 版本兼容策略；
- 文档和接口契约。

### 9.4 适合什么工程

适合：

- 公司有产品线；
- 同一功能要复用到多个硬件；
- 芯片供应风险要求可替换；
- 多团队并行开发；
- 需要维护多个客户定制版本；
- 代码生命周期明显长于单个项目。

不适合：

- 一次性项目；
- 硬件和需求都非常确定的小项目；
- 团队规模很小且没有复用压力；
- 抽象成本大于维护收益。

---

## 10. 不同类型工程的架构侧重点

### 10.1 简单外设控制类

例如 LED、继电器、按键、小型执行器。

推荐：

- 裸机顺序式；
- 中断 + 前后台；
- 简单模块化。

通常不需要：

- RTOS；
- 复杂状态机框架；
- 大量接口抽象；
- 平台化设计。

### 10.2 周期采样和闭环控制类

例如电机控制、电源控制、温控、传感器采样。

推荐：

- Timer / ADC / DMA 快路径；
- 中断驱动；
- 前后台处理；
- 控制算法与硬件访问分离；
- 必要时用轻量调度器。

RTOS 判断：

- 核心控制环路通常不要依赖普通 task 调度；
- 外围通信、UI、存储复杂时，可以引入 RTOS；
- 控制快路径仍应保持短、确定、可测量。

### 10.3 通信协议类

例如 Modbus、CANopen、串口协议、协议转换器。

推荐：

- 中断或 DMA 收发；
- ring buffer；
- 协议解析模块；
- 超时和重发状态机；
- 清晰的 transport / protocol / app 分层。

RTOS 判断：

- 单路简单协议可以不用 RTOS；
- 多路通信并发、协议栈复杂、阻塞等待明显时适合 RTOS；
- 协议状态机比 task 数量更重要。

### 10.4 HMI 和显示类

例如带屏幕、按键、触摸、人机交互页面的设备。

推荐：

- UI task 或 UI 服务；
- 输入事件队列；
- 页面状态机；
- 显示刷新节流；
- GUI 框架和业务逻辑隔离。

RTOS 判断：

- 带图形库、动画、通信、存储时通常适合 RTOS；
- 资源很小的段码屏、简单菜单可以用裸机状态机。

### 10.5 IoT 和联网类

例如 Wi-Fi、BLE、4G、以太网设备。

推荐：

- RTOS 任务化；
- 网络状态机；
- 连接管理；
- OTA 升级；
- 参数和证书存储；
- 本地业务与云侧协议解耦。

RTOS 判断：

- 大多数联网项目适合 RTOS；
- 如果无线模组通过 AT 指令提供完整协议能力，主 MCU 可以简化为前后台 + 状态机；
- 如果协议栈运行在主 MCU 上，RTOS 通常更合适。

### 10.6 安全和高可靠性类

例如工业控制、医疗周边、汽车电子相关设备。

推荐：

- 明确的故障状态机；
- 看门狗策略；
- 关键任务监控；
- 上电自检；
- 参数双备份；
- Bootloader 可靠升级；
- 统一错误码和诊断日志；
- 编码规范和静态检查。

RTOS 判断：

- 能用，但要严格设计优先级、锁、栈、超时和监控；
- 关键安全路径不能只依赖 task 正常调度；
- 故障降级逻辑要能独立、确定地执行。

### 10.7 多产品复用类

例如同一套软件支持多个型号、多个芯片、多个客户版本。

推荐：

- 模块化分层；
- 接口抽象；
- 产品配置；
- 组件化；
- 自动化构建；
- 单元测试或仿真测试。

RTOS 判断：

- 是否使用 RTOS 取决于每个产品的并发复杂度；
- 平台化本身不等于必须使用 RTOS；
- 平台层要允许裸机版本和 RTOS 版本共存。

---

## 11. 按工程体量选择架构

### 11.1 很小型工程

特征：

- 单一功能；
- 外设少；
- 无复杂通信；
- 无长期复用压力；
- 资源很紧。

推荐：

- 裸机顺序式；
- 少量中断；
- 简单模块拆分。

不推荐：

- RTOS；
- 复杂状态机框架；
- 大量抽象接口；
- 平台化目录结构。

### 11.2 小型到中小型工程

特征：

- 有多个外设；
- 有少量异步事件；
- 有固定周期任务；
- 业务逻辑仍然可控。

推荐：

- 中断 + 前后台；
- ring buffer；
- 时间片轮询；
- 简单状态机；
- 初步分层。

RTOS：

- 通常不需要；
- 如果有通信、存储、UI 等阻塞模块，可以重新评估。

### 11.3 中型工程

特征：

- 有协议、参数、存储、诊断、控制策略；
- 有多个状态和异常流程；
- 项目需要长期维护；
- 可能有多人协作。

推荐：

- 模块化分层；
- 事件队列；
- 状态机；
- 协作式调度或有限 task；
- 清晰接口边界。

RTOS：

- 视并发复杂度决定；
- 如果只是状态复杂，不一定需要；
- 如果阻塞和并发明显，可以引入。

### 11.4 中大型工程

特征：

- 通信、控制、UI、存储、升级、诊断并存；
- 多个功能域需要独立优先级；
- 协议栈或 GUI 框架复杂；
- 后续功能会持续增加。

推荐：

- 分层 + RTOS；
- task 按职责域划分；
- 消息队列和事件组；
- 状态机；
- 统一日志、错误码、参数管理。

注意：

- 不要一个小功能一个 task；
- 不要用全局变量绕过队列和接口；
- 不要让锁和临界区扩散到业务各处。

### 11.5 大型或产品线工程

特征：

- 多产品；
- 多平台；
- 多团队；
- 多客户定制；
- 生命周期长；
- 复用和维护成本是主要矛盾。

推荐：

- 平台化；
- 组件化；
- 产品配置化；
- 抽象接口；
- 自动化构建和测试；
- 版本兼容和升级策略。

RTOS：

- 常用但不是平台化的前提；
- 可以按产品裁剪；
- 平台接口要隔离裸机和 RTOS 差异。

---

## 12. 关于 RTOS 的选型判断

### 12.1 可以不用 RTOS 的信号

如果大部分条件成立，可以先不用 RTOS：

- 主循环能清楚表达系统流程；
- 周期任务数量少；
- ISR + 后台能满足实时性；
- 没有复杂阻塞等待；
- 没有多个强并发功能域；
- RAM 资源紧张；
- 团队更熟悉裸机调试；
- 引入 RTOS 只会增加不必要复杂度。

推荐组合：

- 中断快路径；
- ring buffer；
- 主循环调度；
- 软件定时器；
- 状态机；
- 模块化分层。

### 12.2 适合使用 RTOS 的信号

如果出现多项，可以认真考虑 RTOS：

- `while (1)` 里已经很难安排执行顺序；
- 到处是 flag 和全局变量；
- 一处延迟会影响全局；
- 多个功能域互相抢时序；
- 需要表达任务优先级；
- 有明显阻塞等待；
- 使用复杂协议栈或 GUI 框架；
- 需要标准队列、事件组、信号量、互斥锁；
- 项目未来会持续扩展。

### 12.3 使用 RTOS 的基本纪律

使用 RTOS 时建议：

- task 按职责域划分；
- 控制 task 数量；
- 每个 task 有明确输入、输出和周期；
- task 间优先用队列、事件、消息通信；
- 共享资源必须有明确所有权；
- 所有阻塞等待都要有超时；
- 关键任务栈要实测；
- 看门狗要能覆盖 task 卡死场景。

## 13. 关于抽象和抽象类

### 13.1 先区分两个概念

MCU 项目里常常需要“抽象”，但不一定需要“抽象类”。

- 抽象：隔离变化，定义稳定接口；
- 抽象类：用 C++ 继承和虚函数实现接口的一种方式。

真正重要的是边界是否清楚，而不是是否用了某种语言机制。

### 13.2 什么时候需要抽象

这些场景通常需要抽象：

- 同一业务支持不同传感器；
- 同一协议运行在 UART、CAN、TCP 等不同传输层；
- 同一存储服务支持内部 Flash、EEPROM、外部 NOR Flash；
- 上层算法需要脱离硬件做测试；
- 产品线需要替换芯片或板卡；
- 需要 mock/stub 做离线测试。

### 13.3 可选实现方式

常见方式：

- C 的 `struct ops`；
- 函数指针表；
- 句柄 + 回调；
- 编译期配置；
- 链接期替换；
- C++ 抽象类；
- C++ 模板和策略类；
- 组合式对象封装。

### 13.4 什么时候不建议上抽象类

不建议：

- 只有单一实现；
- 项目很小；
- 团队 C++ 经验不足；
- 时序和资源非常敏感；
- 没有测试、替换、复用需求；
- 只是为了让代码看起来有架构。

一句话：

> 抽象层常常有价值，抽象类只是可选工具。

## 14. 选型提问清单

下面这组问题适合在项目立项、架构评审或重构前使用。

### 14.1 工程目标

- 这是一次性 Demo，还是量产项目？
- 项目预计维护多久？
- 是否会有多个硬件版本？
- 是否会有多个客户定制版本？
- 后续需求增长是否明确？
- 当前最重要的是快速交付、稳定量产，还是长期复用？

### 14.2 资源约束

- MCU 的 Flash 和 RAM 余量是多少？
- 主频是否足够？
- 是否有 FPU、DMA、Cache、MPU 等硬件能力？
- 任务栈、缓冲区、协议栈、文件系统会占用多少 RAM？
- 是否允许动态内存？
- 是否有低功耗要求？

### 14.3 实时性和时序

- 最短必须响应时间是多少？
- 哪些逻辑必须在中断或 DMA 快路径完成？
- 控制周期是多少？
- 采样周期是多少？
- 允许最大抖动是多少？
- 哪些任务可以延后处理？
- 哪些事件允许丢弃，哪些不能丢？
- 是否有长耗时操作会阻塞主流程？

### 14.4 外设和数据流

- 有多少路 UART、CAN、SPI、I2C、ADC、PWM？
- 数据是周期采样为主，还是事件触发为主？
- 是否有高速数据流？
- 是否需要 DMA？
- 是否需要双缓冲或 ring buffer？
- 数据从采集到输出经过哪些处理阶段？
- 数据流和控制流是否已经分开？

### 14.5 状态和业务流程

- 设备有哪些运行模式？
- 是否存在待机、运行、故障、升级、恢复等状态？
- 是否有超时、重试、降级、保护流程？
- 是否能画出核心状态图？
- 现在是否已经出现大量嵌套 `if/else`？
- 状态迁移规则是否能被评审和测试？

### 14.6 并发和阻塞

- 是否有多个功能域需要同时运行？
- 是否有通信、控制、UI、存储、诊断并存？
- 哪些模块会阻塞等待事件？
- 是否需要优先级？
- 是否需要队列、信号量、互斥锁？
- 是否已经在裸机里手写了复杂调度逻辑？
- 是否能接受 RTOS 带来的栈开销和调试复杂度？

### 14.7 模块边界

- 驱动层是否和业务层混在一起？
- App 是否直接访问寄存器？
- Driver 是否反向调用 App？
- 模块之间是通过接口通信，还是共享全局变量？
- 每个模块的输入、输出、所有权是否清楚？
- 换一个传感器或通信口，需要改多少上层代码？

### 14.8 复用和移植

- 未来是否要换芯片？
- 未来是否要换板子？
- 是否要支持多个产品型号？
- 是否有可复用的通信、日志、参数、升级组件？
- 产品差异是通过配置表达，还是散落在 `#ifdef` 里？
- 是否需要为模块提供 mock/stub？

### 14.9 调试和可靠性

- 出问题时能否定位到模块、状态、事件？
- 是否有统一日志和错误码？
- 是否有断言或故障码？
- 看门狗喂狗策略是否能发现 task 或主循环卡死？
- 参数是否有校验和备份？
- 升级失败是否能回滚或恢复？
- 是否有上电自检和运行时健康监测？

### 14.10 团队和交付

- 团队熟悉裸机还是 RTOS？
- 团队是否有 C++ 使用经验？
- 是否有足够时间建立抽象和测试？
- 当前最缺的是功能速度、稳定性，还是长期维护能力？
- 架构复杂度是否超过团队可掌控范围？
- 新人能否在合理时间内理解系统结构？

## 15. 快速决策表

| 回答倾向 | 更适合的架构 |
| --- | --- |
| 功能少、资源紧、流程直观 | 裸机顺序式 |
| 有异步事件，但业务不复杂 | 中断驱动 + 前后台 |
| 多个周期任务，但不需要多线程 | 时间片轮询 / 协作式调度 |
| 驱动和业务开始耦合 | 模块化分层 |
| 状态和异常流程复杂 | 事件驱动 / 状态机 |
| 多功能域并发、存在阻塞等待 | RTOS 任务化 |
| 多产品、多芯片、多团队复用 | 平台化 / 组件化 |

也可以按下面顺序做判断：

1. 如果只是单功能小项目，先用裸机顺序式。
2. 如果出现异步事件，引入中断 + 前后台。
3. 如果周期任务变多，引入时间片轮询。
4. 如果代码开始难维护，先做模块化分层。
5. 如果流程和模式变复杂，引入状态机。
6. 如果并发和阻塞变复杂，再引入 RTOS。
7. 如果复用和移植变成主要矛盾，再做平台化和组件化。

---

## 16. 总结

MCU 架构选型可以归纳为一句话：

> 先识别复杂度来源，再选择最小但足够的架构手段。

不同复杂度对应的优先解法是：

- 时序复杂：中断、DMA、Timer、短路径控制；
- 周期复杂：时间片轮询、软件定时器；
- 状态复杂：事件驱动、状态机；
- 模块复杂：分层、接口、依赖治理；
- 并发复杂：RTOS、队列、信号量、任务优先级；
- 复用复杂：组件化、平台化、配置化。

不要把 RTOS 当成所有复杂 MCU 工程的默认答案，也不要把裸机架构看成低级方案。合适的 MCU 架构，应该能在资源、实时性、复杂度、团队能力和产品生命周期之间取得平衡。
