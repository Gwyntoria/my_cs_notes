# 退避策略与幂等处理

## Summary

退避策略 (Backoff Strategy) 解决失败后什么时候再试的问题。幂等处理 (Idempotency) 解决同一个操作被执行多次以后，系统状态还能不能保持正确的问题。

这两件事通常要一起设计。只有退避，没有幂等，重试可能造成重复扣款、重复下单、重复发消息、重复写数据库。只有幂等，没有退避，大量客户端在故障恢复时同时重试，会把刚恢复的服务再次打垮。

对任何有副作用的操作，可以按这条原则处理：**客户端有限、分散地重试；服务端识别重复请求，并返回稳定结果**。

## 失败不只是一种状态

网络调用里最麻烦的地方是：客户端超时，不代表服务端失败。

一次请求可能落在几种不同状态里：

| 场景 | 服务端状态 | 客户端看到的结果 | 处理方式 |
| :--- | :--- | :--- | :--- |
| 请求没有发出去 | 未执行 | 连接失败 | 可以重试 |
| 请求到达前连接断开 | 未执行 | 超时或连接断开 | 可以重试 |
| 服务端执行失败 | 未提交或已回滚 | `5xx` 错误 | 按错误类型判断是否重试 |
| 服务端执行成功，响应丢失 | 已提交 | 超时 | 只能靠幂等或查询确认 |
| 服务端仍在执行 | 状态未知 | 超时 | 等待、查询或幂等重试 |

退避策略只能控制重试节奏，不能证明业务没有重复执行。幂等处理能兜住重复请求，但不能控制流量冲击。支付、订单、消息投递、库存扣减、设备控制这类系统，都需要把二者放在同一个方案里看。

## 退避策略控制重试节奏

退避策略用于决定失败后等多久再重试。常见做法有几种：

| 策略 | 说明 | 风险 |
| :--- | :--- | :--- |
| 固定间隔重试 | 每隔固定时间重试一次 | 大量客户端容易同步重试 |
| 线性退避 | 每次重试间隔按固定步长增加 | 增长较慢，高峰期压力仍然大 |
| 指数退避 | 每次重试间隔按倍数增加 | 没有上限时等待时间会失控 |
| 指数退避 + 抖动 | 在指数退避基础上加入随机扰动 | 实现稍复杂，但更适合真实系统 |

指数退避 (Exponential Backoff) 的典型形式：

```text
delay = min(base_delay * 2^retry_count, max_delay)
```

加入抖动 (Jitter) 后：

```text
delay = random(0, min(base_delay * 2^retry_count, max_delay))
```

抖动的作用是打散重试时间。服务端故障恢复、网络恢复、设备批量上线时，如果所有客户端都按同一个间隔重试，很容易形成第二波流量峰值。

## 幂等处理兜住重复副作用

幂等是指同一个操作执行一次和执行多次，对系统产生的最终效果一致。

HTTP (Hypertext Transfer Protocol) 中，`GET`、`PUT`、`DELETE` 通常被设计为幂等请求；`POST` 常用于创建资源或提交动作，通常不是天然幂等。但真实业务里，是否幂等不只由 HTTP 方法决定，而是由业务语义决定。

- `PUT /users/123/email` 把邮箱设置为同一个值，通常是幂等的。
- `POST /orders` 创建订单，如果不带幂等键，通常不是幂等的。
- `POST /payments` 扣款，即使用 `POST`，也必须通过幂等键做成业务幂等。
- `DELETE /files/abc` 删除文件，第一次返回成功，后续重复删除可以返回成功或已不存在，但不能变成系统错误。

更通用的判断方式是看操作有没有副作用，以及重复执行副作用会不会改变业务结果。读请求一般风险小；创建、扣减、发送、转账、控制设备这类操作，默认都要有幂等设计。

## 两者如何配合

以创建订单为例，一个稳妥的流程是：

1. 客户端生成全局唯一的幂等键，比如 `Idempotency-Key`。
2. 客户端发送创建订单请求，带上幂等键、业务参数和请求超时。
3. 服务端先检查幂等记录是否存在。
4. 如果幂等键不存在，服务端创建幂等记录，状态标记为 `PROCESSING`。
5. 服务端执行业务逻辑，比如校验、锁库存、创建订单、写出站消息。
6. 业务提交成功后，服务端把幂等记录更新为 `SUCCEEDED`，并保存响应摘要或资源标识 (Identifier, ID)。
7. 客户端收到成功响应后结束流程。
8. 如果客户端超时或遇到可重试错误，按指数退避 + 抖动重试。
9. 服务端收到相同幂等键时，返回已保存的成功结果，或返回当前处理状态。

伪代码：

```text
client:
    key = generate_idempotency_key()

    for attempt in 0..max_retries:
        response = call_api(key, payload, timeout)

        if response.success:
            return response

        if response.is_non_retryable:
            fail(response)

        sleep(backoff_with_jitter(attempt))

server:
    record = find_idempotency_record(key)

    if record.status == SUCCEEDED:
        return record.saved_response

    if record.status == PROCESSING:
        return current_processing_response

    create_record(key, payload_hash, PROCESSING)

    result = execute_business_operation()
    save_record(key, payload_hash, SUCCEEDED, result)

    return result
```

这里最容易出错的是客户端每次重试都生成新的幂等键。这样服务端看到的是多次新请求，幂等机制就失效了。

## 服务端幂等如何设计

### 幂等键要对应业务意图

幂等键应该由客户端生成，并能稳定标识一次业务意图。

| 场景 | 幂等键 |
| :--- | :--- |
| 支付 | `merchant_id + payment_request_id` |
| 下单 | `user_id + client_order_id` |
| 消息发送 | `producer_id + message_id` |
| 设备控制 | `device_id + command_id` |

幂等键不应该只依赖时间戳或随机数。随机数可以参与生成，但服务端需要明确知道什么算同一个业务请求。

### 同一个键必须校验参数

同一个幂等键重复请求时，请求参数必须一致。服务端通常需要保存 `payload_hash`。后续请求带相同幂等键但参数不同，应返回参数冲突错误，不能当成新业务继续执行。

```text
Idempotency-Key: order-123
payload_hash: hash(user_id, sku_id, quantity, price)
```

如果第一次请求是买 1 件商品，第二次同一个幂等键却变成买 2 件，服务端必须拒绝。

### 幂等记录需要状态机

幂等记录不应该只有存在和不存在两种状态。真实系统至少需要区分：

| 状态 | 含义 | 重复请求处理 |
| :--- | :--- | :--- |
| `PROCESSING` | 请求正在处理 | 返回处理中、建议稍后查询，或短暂等待 |
| `SUCCEEDED` | 请求已经成功 | 返回第一次成功的结果 |
| `FAILED_RETRYABLE` | 临时失败，可重试 | 允许同幂等键再次尝试 |
| `FAILED_FINAL` | 最终失败，不应重试 | 返回失败结果 |
| `EXPIRED` | 幂等记录过期 | 按业务规则拒绝或要求发起新请求 |

状态机要和事务边界一起设计。最危险的情况是业务已经提交成功，但幂等记录没有保存成功。常见做法是把业务结果和幂等记录放在同一个事务中，或者通过唯一约束、出站表 (Outbox) 和补偿任务保证最终一致。

### 唯一约束是最后防线

数据库唯一约束是幂等处理的最后防线。不要只依赖应用层的先查再写，因为并发请求可能同时通过查询。

```sql
CREATE UNIQUE INDEX uniq_order_client_id
ON orders(user_id, client_order_id);

CREATE UNIQUE INDEX uniq_idempotency_key
ON idempotency_records(scope, idempotency_key);
```

当并发插入发生冲突时，服务端应该把唯一键冲突转换成幂等结果查询，而不是直接返回数据库错误。

## 客户端重试如何设计

### 只重试可重试错误

不是所有失败都应该重试。

| 错误类型 | 是否重试 | 说明 |
| :--- | :--- | :--- |
| 连接失败 | 可以 | 请求可能未到达 |
| 读超时 | 可以，但必须幂等 | 服务端可能已经执行成功 |
| `429 Too Many Requests` | 可以 | 遵守 `Retry-After` |
| `500`、`502`、`503`、`504` | 通常可以 | 需要结合业务和接口语义 |
| `400`、`401`、`403` | 通常不重试 | 请求或权限问题 |
| 参数校验失败 | 不重试 | 重试不会改变结果 |
| 余额不足、库存不足 | 不重试或转业务流程 | 属于业务失败 |

读超时要特别小心。它只说明客户端没有读到响应，不说明服务端没有执行。对有副作用的接口，读超时应该按状态未知处理。

### 重试必须有边界

重试需要同时限制次数和耗时：

- 最大重试次数，比如 3 次。
- 最大总耗时，比如 10 秒。
- 最大单次等待时间，比如 2 秒。
- 请求级超时，比如每次 RPC (Remote Procedure Call) 调用 1 秒。
- 调用链路总截止时间，比如用户请求最多等待 5 秒。

如果没有总截止时间，多个服务层层重试会放大延迟和流量，最后形成重试风暴。

### 避免调用链重试放大

一个用户请求可能经过 API (Application Programming Interface) 网关、业务服务、库存服务、支付服务和消息服务。如果每一层都重试 3 次，底层服务在最坏情况下会收到被放大的请求。

常见控制方式：

- 只在最靠近失败点的一层重试。
- 上游传递统一的 `deadline`，不要每层重新计算超时。
- 非关键路径放到异步队列，不阻塞主请求。
- 对热点资源做限流和熔断。
- 在请求上下文里记录 `attempt_count`，避免重复重试。

## 真实系统里的用法

### 支付扣款

支付系统必须假设客户端超时但扣款成功会发生。

常见做法：

- 商户侧生成 `payment_request_id`。
- 支付网关以 `merchant_id + payment_request_id` 做唯一键。
- 同一幂等键重复请求时，返回同一笔支付单结果。
- 客户端超时后先查询支付状态，查询不到再按退避策略重试。
- 扣款、支付单状态、幂等记录和消息投递需要有一致性保障。

支付接口不能只靠客户端不要重复点击来保证正确性。

### 订单创建

订单系统的重点是避免重复下单和重复扣库存。

常见做法：

- 客户端生成 `client_order_id`。
- 服务端对 `user_id + client_order_id` 建唯一约束。
- 库存扣减使用订单 ID 或库存流水 ID 做幂等。
- 创建订单后通过 Outbox 写消息，异步通知仓储、优惠券、积分等系统。
- 客户端创建超时后，用 `client_order_id` 查询订单状态。

### 消息投递

消息系统通常追求至少一次投递 (At-Least-Once Delivery)，这意味着消费者必须具备幂等处理能力。

常见做法：

- 生产者为消息生成全局唯一 `message_id`。
- 消费者保存已处理消息表，或在业务表中使用唯一业务键。
- 消费失败时按退避策略重新投递。
- 多次收到同一消息时，消费者直接返回成功，避免消息反复堆积。

**消息中间件保证不丢时，经常会牺牲不重复。业务侧必须能处理重复消息。**

### 物联网系统

物联网 (Internet of Things, IoT) 系统更依赖退避和幂等的组合。设备侧经常处在弱网、断电重启、低功耗休眠、网关切换、批量上线这些条件下。云端看见的现象通常是重复上报、消息乱序、响应丢失、设备集体重连。

常见场景可以这样处理：

| 场景 | 退避策略 | 幂等处理 |
| :--- | :--- | :--- |
| 设备重连云端 | 指数退避 + 抖动，避免大量设备同时上线 | 用 `device_id` 恢复会话和订阅状态 |
| 遥测数据上报 | 上传失败后有限重试，低功耗设备要限制总耗时 | 数据带 `message_id`、采样时间、序列号，云端去重 |
| 云端下发控制命令 | 命令超时后退避重发或查询状态 | 命令带 `command_id`，设备重复收到只返回上次结果 |
| 离线缓存补传 | 网络恢复后分批上传，批次之间加入抖动 | 云端按设备、采样序号、数据窗口去重 |
| 空中下载 (Over-the-Air, OTA) 升级 | 分片下载失败后退避重试 | 用版本号、分片编号、校验和保证重复下载安全 |
| 网关转发 | 转发失败后退避，避免阻塞所有设备消息 | 网关和云端都按 `message_id` 去重 |

OTA 升级和设备控制尤其要谨慎。`toggle_power()` 这种切换命令不是幂等的，重复执行两次可能回到原状态。更稳的接口是 `set_power(on)`，重复执行多次，最终状态仍然是开机。门锁、电机、继电器、升级任务都尽量表达成目标状态，而不是表达成切换动作。

一个设备控制命令可以设计成这样：

```json
{
  "device_id": "dev-001",
  "command_id": "cmd-20260505-001",
  "command_type": "set_power",
  "target_state": "on",
  "expire_at": "2026-05-05T12:00:00Z",
  "sequence": 1024
}
```

设备侧处理时，先检查 `command_id` 是否处理过，再检查命令是否过期，然后比较 `sequence` 是否比当前状态版本更新。执行成功后保存命令 ID 和执行结果。重复收到同一命令时，返回上次结果，不重复执行副作用。

物联网系统还要考虑设备资源限制。电池设备不能无限重试；设备本地存储有限，不能永久保存所有命令 ID；设备时钟可能不准，不能只靠本地时间判断命令是否过期。常见做法是保存最近 N 条命令 ID，结合序列号、云端时间和过期时间一起判断。

## 设计检查清单

设计一个需要重试的接口时，可以按下面这些问题过一遍：

1. 这个操作是否有副作用？
2. 客户端超时时，服务端是否可能已经执行成功？
3. 客户端是否有稳定的业务 ID 可以作为幂等键？
4. 同一个幂等键但参数不同，服务端如何处理？
5. 幂等记录保存多久，过期后如何处理？
6. 幂等记录和业务结果是否在同一事务或同一一致性机制下？
7. 哪些错误允许重试，哪些错误必须立即失败？
8. 重试次数、单次超时、总截止时间分别是多少？
9. 是否加入抖动，避免故障恢复时同步重试？
10. 调用链上是否存在多层重试放大？
11. 重复请求返回的是上次结果，还是当前状态？
12. 幂等命中、重试次数、超时、冲突、最终失败是否有监控指标？
13. 如果是 IoT 设备，是否考虑低功耗、离线缓存、命令过期、消息乱序和设备重启？

## 常见错误

- 只做客户端防抖，不做服务端幂等。
- 把随机请求 ID 当成幂等键，但没有绑定业务意图。
- 客户端每次重试都生成新的幂等键。
- 服务端先执行业务，再写幂等记录，中途失败后无法识别重复请求。
- 重试所有错误，包括参数错误、权限错误和业务失败。
- 没有抖动，导致大量客户端在同一时间重试。
- 每一层服务都自动重试，造成底层服务流量放大。
- 幂等记录永久保存，导致存储无限增长。
- 幂等记录过期太快，客户端重试窗口还没结束就失效。
- 同一个幂等键参数不一致时仍继续执行。
- IoT 设备只保存当前状态，不保存命令 ID 或序列号，导致重复命令被重复执行。

## 实践原则

1. 先设计幂等，再设计重试。
2. 对有副作用的 `POST` 请求，默认要求幂等键。
3. 读超时按状态未知处理。
4. 退避策略使用指数退避 + 抖动，并设置最大重试次数和总截止时间。
5. 服务端用唯一约束兜底，不只依赖应用层判断。
6. 幂等键要绑定业务语义和参数摘要，避免同键不同义。
7. 成功结果要可复用，重复请求应返回稳定响应。
8. 临时失败和最终失败要区分，不能把所有失败都缓存成永久失败。
9. 调用链只在少数明确位置重试，避免层层放大。
10. 对状态未知的长流程，优先提供查询接口。
11. 对 IoT 设备，优先设计目标状态命令，比如 `set_power(on)`，少用 `toggle_power()` 这类切换命令。
